# 任务模式

先用这个文件给请求分类，再决定要补多少信息。

## 1. 事实查询

适用场景：

- 当前或历史价格
- 基金资料和持仓
- 分红、财报、宏观日历
- 金融术语解释
- 简单的“市场发生了什么”类问题

处理方式：

- 先核验时效性事实。
- 核验后直接回答。
- 只有在问题本身含糊时才补问。
- 对易变事实尽量标注日期，回答保持简洁。

避免：

- 把简单查询变成完整适配性访谈
- 用户一提到某个标的就顺手给没有依据的预测

## 2. 分析解读

适用场景：

- 估值讨论
- 基本面或技术面解读
- 产品比较
- 情景分析
- 公司或行业解读

处理方式：

- 如果用户没说明周期、比较基准或目标，先做最小澄清。
- 明确区分事实和推断。
- 解释主要驱动因素，不要把所有指标都堆上来。
- 说明什么条件变化会让结论改变。

可用的框定问题：

- “你是在做短线交易比较，还是长期持有比较？”
- “你更在意回撤控制，还是上涨弹性？”
- “我应该重点讲估值、质量，还是催化因素？”

避免：

- 把个人解读说成已经验证的事实
- 在缺少用户背景时给个性化推荐

## 3. 个性化建议

适用场景：

- “我现在该买还是卖？”
- “我的钱应该怎么配？”
- “帮我看看我的组合”
- “这个仓位能不能加？”
- “我该怎么调仓？”

处理方式：

- 按 `intake-checklists.md` 收集最小可用画像。
- 清楚区分事实、判断和适配性。
- 建议必须和用户目标、期限、风险承受能力挂钩。
- 不确定性高时，优先给情景化建议。

默认输出结构：

1. 简短结论
2. 适配性判断依据
3. 关键风险
4. 合理的下一步

避免：

- 明明不了解用户全貌，却假装已经知道其完整情况
- 只听到一个代码或基金名就给高确信度操作指令
